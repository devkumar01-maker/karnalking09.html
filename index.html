Module 1: Fundamentals of Bug Bounty Report Writing

Why Good Reports Matter

  • First Impression: Your report is often the first interaction with the security team
  • Clear Communication: Technical findings must be understandable to both technical and non-technical audiences
  • Faster Triage: Well-structured reports get processed quicker
  • Higher Bounties: Quality reports often receive better rewards
  • Reputation Building: Good reporters get invited to private programs

Key Principles

  1. Clarity over Cleverness: Simple, clear language wins
  2. Completeness: Include all necessary information
  3. Reproducibility: Anyone should be able to follow your steps
  4. Impact Focus: Clearly explain business risk
  5. Professional Tone: Maintain respectful, constructive communication

Module 2: Report Structure and Components

Essential Report Structure

  1. Title/Summary
  2. Vulnerability Classification
  3. Affected Asset/URL
  4. Vulnerability Description
  5. Steps to Reproduce
  6. Proof of Concept (PoC)
  7. Impact Assessment
  8. Remediation Recommendations
  9. Additional Information/References

Title Examples

  • "SQL Injection in user search functionality leads to database compromise"

Module 3: Writing Best Practices

Language and Tone

‚úÖ DO:

  • Use professional language
  • Be respectful and constructive
  • Write in third person
  • Use technical terms correctly
  • Be concise but complete

‚ùå DON'T:

  • Use aggressive language
  • Oversell the impact
  • Include unnecessary personal details
  • Use slang or informal language
  • Make demands or threats

Technical Writing Tips

  1. Use Active Voice: "The application accepts malicious input" vs "Malicious input is accepted"
  2. Be Specific: Include exact error messages, response codes
  3. Use Consistent Formatting: Code blocks, bullet points, numbering
  4. Include Context: Environment details, browser versions, etc.

Report Example 1: SQL Injection

Title: SQL Injection in User Search Functionality Leads to Database Disclosure Vulnerability Type: CWE-89: SQL Injection CVSS Score: 9.1 (Critical) Affected Asset: https://target.com/search.php

Executive Summary:

A SQL injection vulnerability exists in the user search functionality of target.com. An authenticated attacker can exploit this vulnerability to extract sensitive information from the database, including user credentials, personal information, and administrative data.

Steps to Reproduce:

  1. Login to the application using valid credentials
  2. Navigate to https://target.com/search.php
  3. Intercept the request using Burp Suite or similar proxy tool
  4. Modify the search query to include SQL injection payload
  5. Forward the request and observe the response
  6. Verify data extraction in the search results page

‚ùå Bad: "This will destroy the entire company"

Module 7: Advanced Report Writing Techniques

Writing for Different Audiences

Technical Audience (Developers/Security Engineers):

  • Include detailed technical explanations
  • Provide code examples and fixes
  • Reference specific frameworks and libraries
  • Include HTTP requests/responses

Business Audience (Management/Product Owners):

  • Focus on business impact
  • Use non-technical language when possible
  • Quantify risks where possible
  • Provide clear recommendations

Handling Complex Vulnerabilities

For complex attack chains:

  1. Break down into clear phases
  2. Explain the logical flow
  3. Show cumulative impact
  4. Provide individual and combined fixes

Video Proof of Concepts

When to use video PoCs:

  • Complex multi-step attacks
  • UI-based vulnerabilities
  • Race conditions
  • Real-time exploitation

Video guidelines:

  • Keep it concise (2-5 minutes)
  • Use clear narration
  • Show each step clearly
  • Include timestamps in description
